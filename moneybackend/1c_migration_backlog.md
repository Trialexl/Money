# Бэклог: паритет между `1c/` и Django `money`

## Область работ

Этот бэклог описывает расхождения между выгрузкой конфигурации 1С в `1c/`, расширением обмена в `1c/расширения/` и текущей реализацией в Django-приложениях `money` и `users`.

Базовое допущение: целью является паритет бизнес-логики и аналитики, а не обязательное совпадение механики платформы 1С.

Отдельно зафиксировано:

- момент записи регистров не является целью паритета
- наличие отдельного статуса проведения/отмены проведения не является целью паритета
- если бизнес-результат совпадает, Django может оставаться на своей модели сохранения данных

## Обозначения приоритетов

- `P0` критичное расхождение в бизнес-логике
- `P1` важный функциональный пробел
- `P2` полезное выравнивание поведения или UX
- `P3` необязательная доработка или фиксация договоренностей

### 13. AI-сервис естественного ввода операций из скриншотов и текста

- Статус: в работе

- Проблема:
  Сейчас операции создаются вручную или через синхронизацию.
  Нет сервиса, который принимает банковский скриншот или свободный текст пользователя, извлекает смысл операции и создает `Receipt`, `Expenditure` или `Transfer` в Django.
- Цель:
  Добавить AI-сервис, который:
  - принимает скриншот из банка
  - принимает свободный текст операции
  - принимает текстовые запросы на остатки

  - подбирает наиболее вероятную `CashFlowItem` там, где статья нужна
  - создает в системе `Receipt`, `Expenditure` или `Transfer`
  - по запросу возвращает остатки по кошелькам без создания операции
  - доступен из веб-приложения и через Telegram-бота
- Базовый подход:
  - использовать multimodal LLM
  - провайдер по умолчанию: `OpenRouter`
  - модель по умолчанию через него: `Gemini`
  - архитектурно сделать провайдер LLM сменяемым, чтобы позже можно было подключить другой backend
- Предлагаемое поведение:
  - вход:
    - изображение скриншота + опционально `wallet`
    - свободный текст, например:
      - `приход сбербанк 10000`
      - `перевод сбербанк альфа 20000`
      - `какой остаток на сбербанке`
      - `остатки по кошелькам`
  - каналы входа:
    - web upload в приложении
    - отправка изображения или текста в Telegram-бот
  - выход LLM: строго структурированный JSON
  - после распознавания backend либо:
    - создает соответствующий документ:
      - `Receipt`
      - `Expenditure`
      - `Transfer`
    - либо возвращает ответ по остаткам:
      - конкретного кошелька
      - всех кошельков
  - если уверенность в типе операции, кошельке или статье недостаточна, сервис либо:
    - создает draft/preview на подтверждение
    - либо возвращает уточняющий вопрос пользователю
- Что нужно определить и реализовать:
  - endpoint загрузки скриншота
  - endpoint текстового AI-ввода операции
  - отдельный webhook/handler для Telegram-бота
  - временное хранение изображения или прямой одноразовый разбор
  - prompt и JSON schema для LLM-ответа
  - классификацию intent:
    - `create_receipt`
    - `create_expenditure`
    - `create_transfer`
    - `get_wallet_balance`
    - `get_all_wallet_balances`
  - словарь признаков и alias для распознавания кошельков по тексту (`сбербанк`, `альфа`, и т.д.)
  - словарь признаков для автоподбора `CashFlowItem`
  - fallback, если статья, кошелек или тип операции не распознаны
  - правила, когда статья обязательна:
    - для `Expenditure` обязательна
    - для `Receipt` обязательна
    - для `Transfer` зависит от `include_in_budget`
  - логику выбора `wallet`, если у пользователя несколько банковских кошельков
  - привязку Telegram-пользователя к пользователю Django
  - единый domain service, чтобы web и Telegram использовали один и тот же pipeline распознавания и создания расхода
  - поддержку уточняющих сценариев в Telegram:
    - не распознан кошелек
    - не распознан тип операции
    - не найдена уверенная статья
    - не удалось однозначно понять, нужен остаток одного кошелька или всех
  - защиту от дублей по одному и тому же скриншоту/операции
